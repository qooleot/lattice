# Projections

`spec.lat` is the single hand-editable source of truth. Everything else that describes a context
or a workspace is a **generated projection** — rendered fresh on every
[`apply`/`sync`](editing.md), plus on `engine emit` and the workspace-level `engine docs` command.
None of these files are meant to be hand-edited; each one that lattice writes carries a
do-not-edit header, and the next apply overwrites it unconditionally.

## Per-spec projections

Alongside `specs/<name>/spec.lat`, a successful `apply`/`sync`/`emit` writes:

- **`specs/<name>/spec.prose.md`** — the human-readable English rendering: the context's doc,
  each entity/aggregate/event with its fields, and every invariant (explicit *and*
  [derived](derived-invariants.md)) with its English sentence and provenance lineage (which
  witnesses, which verdicts, which date). This is where a derived invariant's "implied by
  structure" note actually renders — the gate's provenance-visibility requirement is satisfied
  here, not in `spec.lat` itself, since derived invariants are never printed as code.
- **`specs/<name>/spec.diagrams.md`** — one page embedding every diagram for this context inline:
  a domain diagram plus one statechart per aggregate region that has a `machine`. An aggregate
  with no `machine` block simply contributes no statechart section — projections generate what's
  modeled, nothing more.
- **`specs/<name>/diagrams/*.mmd`** — the same diagrams as standalone Mermaid source files, one
  per diagram (`CD_<Context>.mmd` for the domain diagram, `SD_<Aggregate>_<Region>.mmd` per
  region), renderable on their own by tooling that only reads `.mmd` files.

Each generated Markdown file opens with `<!-- generated by lattice; do not edit -->`; each `.mmd`
file opens with `%% generated by lattice; do not edit`.

## Workspace-level projection

When a `.lat` file lives inside a workspace (a `context-map.lat` sits at its grandparent
directory), `apply` also recompiles the workspace's diagram projection as a side effect, and the
standalone `engine docs --workspace <dir>` command does the same on demand:

- **`context-map.generated.md`** — the same do-not-edit-headed Markdown convention, embedding the
  context-map flowchart plus a `## Relationships` bullet list (each relationship's kind and,
  if present, its `///` doc).
- **`diagrams/context-map.mmd`** — the standalone Mermaid source for that same flowchart.

Diagnostics from a broken *sibling* member context are reported alongside the workspace hook's
result but never block the `apply` that triggered them — the edit that was actually requested
already reconciled successfully by the time the workspace hook runs.

## Why regeneration matters

Because every one of these files is rebuilt from the canonical model on every apply/sync/emit/docs
run, none of them can drift from `spec.lat`: an engineer can always regenerate the whole set by
re-running `apply` (or `docs` for the workspace-level ones), and none of them are inputs to
anything — they are always outputs. This is also why doc comments live only in `.lat` and not in
the generated files directly: the `///` text is the input, and it is rendered into `spec.prose.md`
and the diagram pages, not the other way around.

## Semantic Rules

- Every generated file is fully overwritten on each successful `apply`/`sync`/`emit`/`docs` run;
  no projection content is ever merged with prior file contents.
- An aggregate with no `machine` block contributes no statechart section to `spec.diagrams.md`
  and no `SD_*.mmd` file.
- The workspace-level projection only runs when a `context-map.lat` is found at the expected
  location relative to the applied file; a standalone context outside any workspace never
  triggers it.
- A refused `apply` (any pipeline step rejects) writes none of these files — the atomicity
  guarantee from [editing](editing.md) covers projections too.
